00How to use this

A reference for recall, not a textbook to read once.

Each part moves from foundational concept → design decision → the trade-off question an interviewer will press on. Design rounds reward reasoning about trade-offs, not term recall — so for every concept, practice saying the why and the alternative out loud.

Every part ends with a Readings panel: a tight, curated list (not a link dump) of the best free source to actually learn that topic, with a one-line note on why it's worth your time. Pair this document with the Study Itinerary, which sequences when to read and what to build.

Orientation Three things show up in nearly every interview and nearly every engagement: declaring grain (Part 3), the lakehouse vs warehouse vs lake distinction (Part 1), and migration reconciliation (Part 2 — your differentiator). If you over-invest anywhere, over-invest there.

01Data Platform & Warehouse Modernization

The architectures, storage formats, and engines you're hired to move clients toward.

Core architectures — know the lineage

ArchitectureWhat it isWhen to useTrade-offs
Data WarehouseSchema-on-write, structured, optimized for BI/SQLGoverned reporting, finance, defined KPIsRigid; costly for raw/unstructured data
Data LakeSchema-on-read, raw files (Parquet/JSON) on object storageCheap raw storage, ML, exploration"Data swamp" risk; weak governance/ACID
LakehouseLake storage + warehouse semantics (ACID table format on object store)Unify BI + ML on one copy of dataNewer tooling maturity; table-format lock-in
Data MeshOperating model: domain-owned data products, federated governanceLarge orgs, many domainsOrg maturity required; not a tech you install
Data FabricMetadata-driven integration layer across sourcesHeterogeneous sprawl, virtualizationComplexity; tooling-heavy
The distinction interviewers wantLakehouse is an architecture pattern; Data Mesh is an organizational/ownership pattern. They're orthogonal — you can run a mesh on lakehouse tech.

Table formats — the lakehouse enablers

Apache Iceberg, Delta Lake, Apache Hudi are open table formats that add ACID transactions, schema evolution, time travel, and partition evolution on top of files in object storage.

metadata / manifest filessnapshot isolationtime travelcompaction / small-file problemhidden partitioning (Iceberg)copy-on-write vs merge-on-read
Probe"Why a table format instead of raw Parquet?" → ACID, concurrent writers, schema evolution, time travel, and decoupling the query engine from storage.

Storage & file concepts

  • Columnar formats (Parquet, ORC): compression, predicate pushdown, column pruning.
  • Row vs columnar: OLTP (row) vs OLAP (columnar) — the fundamental analytics storage choice.
  • Partitioning (by date/key) vs clustering / Z-ordering / sort keys (data skipping).
  • Small-file problem and compaction; target ~128MB–1GB files for analytics scans.

Compute & warehouse engines

  • Separation of storage and compute — the defining cloud-DW shift; Snowflake popularized it. Lets you scale each independently and isolate workloads.
  • MPP fundamentals: shuffle, broadcast vs hash joins, partition pruning, data skew.
  • Snowflake: virtual warehouses, micro-partitions, automatic clustering, time travel, zero-copy cloning, multi-cluster warehouses for concurrency, RBAC.
  • Databricks: Spark execution model, Photon engine, Unity Catalog (governance), Delta Lake, clusters vs SQL warehouses, medallion architecture.
  • BigQuery / Redshift / Synapse: BigQuery = serverless slots; Redshift = node-based MPP; Synapse = MS ecosystem.
Design prompts to rehearse "Migrate a legacy on-prem EDW (Teradata/Oracle/SQL Server) to a cloud lakehouse." · "Warehouse vs lakehouse — how do you choose for a client?" · "Size compute / control cost." · "Handle concurrency for 500 BI users." (multi-cluster compute, result caching, extracts vs live)
Readings — Platform & Warehouse
  • Databricks — "What is a Data Lakehouse?" + "Medallion Architecture" glossary pages. Clearest free explanation of the lakehouse thesis and the Bronze/Silver/Gold layering you'll reuse everywhere.
  • Snowflake docs — "Key Concepts & Architecture" (virtual warehouses, micro-partitions, separation of storage/compute). The canonical mental model for modern cloud DW; even if you target BigQuery, this framing is what interviewers speak.
  • Apache Iceberg docs — "Iceberg Table Spec" overview + the "why Iceberg" intro. Skim for the metadata/manifest/snapshot model; that's the table-format vocabulary.
  • Google Cloud — "BigQuery explained: storage overview" blog series. Your cloud target's internals — slots, columnar storage, partitioning/clustering.
  • Book (skim) — Fundamentals of Data Engineering (Reis & Housley), Ch. 3 & 6. The single best survey of the architecture landscape if you want one durable source.

02Data Migration & Validation

Your stated specialty. Go deeper here than anywhere else — it's both an interview edge and a sellable Eigenbasis methodology.

Migration strategy patterns

  • Lift-and-shift (rehost) vs re-platform vs re-architect (refactor).
  • Big-bang vs phased / incremental vs parallel run.
  • Strangler pattern: incrementally route consumers off the legacy system as the new system proves out.
  • Dual-write / shadow-write to compare old vs new during transition.

The migration lifecycle — be able to whiteboard this

  1. Discovery & assessment — source inventory, lineage, profiling, volumetrics, dependency mapping.
  2. Schema mapping & conversion — type mapping, source quirks, surrogate keys.
  3. Initial (historical) load — bulk/backfill.
  4. CDC / incremental sync — keep target current during the cutover window.
  5. Validation & reconciliation — see below.
  6. Cutover — freeze, final delta, switch consumers.
  7. Rollback plan — always have one.
  8. Decommission legacy.

Validation & reconciliation — go deep

  • Row count reconciliation (control totals).
  • Aggregate / checksum reconciliation — SUM/MIN/MAX/hash of columns; column-level hashing (MD5/SHA over concatenated rows).
  • Row-level diffing — minus/except queries, hash-comparison per primary key.
  • Schema validation — types, nullability, constraints, precision/scale drift.
  • Referential integrity across migrated tables.
  • Parallel-run validation — run old & new for N cycles, compare outputs.
  • Sampling vs full reconciliation — cost/coverage trade-off; tiered (full counts + sampled deep checks).
  • Tooling: custom SQL frameworks, dbt tests, Great Expectations, data-diff tools.
Failure classes to name on sighttruncation · rounding · charset/collation · null vs empty string · date/timezone shifts · duplicate explosion from bad joins · late-arriving data. Naming these unprompted is the single strongest reconciliation signal you can give.

Probe — your sweet spot"How do you validate a migration moved data correctly?" Walk the tiered ladder: row counts → column checksums → row-level hash diff on PK → schema/type validation → parallel run. Then name two failure classes you'd deliberately test for.
Readings — Migration & Validation
  • Debezium docs — "CDC connectors" overview + the FAQ on snapshot vs streaming. The reference implementation of log-based CDC; gives you the vocabulary (snapshots, tombstones, offsets) without needing to run it.
  • Datafold blog — posts on data diffing and validating data migrations. The clearest practitioner write-ups on reconciliation strategy specifically; mirrors the tiered approach above.
  • Martin Fowler — "StranglerFigApplication" + "ParallelChange / expand-contract." Canonical articulation of incremental migration safety; reframe these patterns for data and you sound senior.
  • Great Expectations docs — the "core concepts" tour (Expectations, Suites, Checkpoints). You'll use GE for source-to-target validation; learn the object model here.
  • AWS / GCP migration whitepapers — any cloud's "database migration best practices" guide. Vendor-neutral on the lifecycle; read for the cutover/rollback framing, ignore the product pitch.

03Analytics Engineering & the Modern Data Stack

The dbt-centered core of most billable work — and the densest interview territory.

The ELT paradigm

ETL vs ELT: the modern stack pushes transformation into the warehouse (ELT) because cloud compute is cheap and elastic. The stack layers:

ingestion (Fivetran/Airbyte/custom) → storage+compute (Snowflake/Databricks/BigQuery) → transformation (dbt) → semantic layer → BI/activation (Power BI/Looker/Tableau) ↑ orchestration (Airflow/Dagster) ↑ observability + catalog (cross-cutting)

dbt — the centerpiece (know cold)

  • Models (SELECT materialized as table/view), ref() (builds the DAG), source().
  • Materializations: view, table, incremental, ephemeral, materialized view, snapshots (SCD2).
  • Incremental models: is_incremental(), unique_key; strategies — append, merge, delete+insert, insert_overwrite; late-arriving data.
  • Tests: generic (unique, not_null, accepted_values, relationships) + singular + packages (dbt_utils, dbt_expectations).
  • Macros / Jinja for DRY, parameterized logic. Seeds, exposures, metrics, semantic models.
  • Docs & lineage (auto-generated DAG), source freshness, packages.
  • dbt build vs run/test/seed/snapshot; slim CI / state:modified (deferral, build-only-changed); dev/prod targets; CI/CD.

Dimensional modeling (Kimball) — interviewers love this

  • Star schema (fact + dimensions) vs snowflake schema (normalized dims).
  • Fact tables: transaction, periodic snapshot, accumulating snapshot; measures are additive / semi-additive / non-additive.
  • Dimensions: conformed, degenerate, role-playing, junk.
  • Grain — the single most important interview word. Always declare the grain first.
  • SCD: Type 0/1/2/3/4/6 — be fluent in Type 1 (overwrite) vs Type 2 (history rows with effective/expiry dates + current flag).
  • Surrogate vs natural keys · bridge tables (M:N) · factless fact tables.

Other modeling paradigms — name them, know when

ParadigmShapeBest fit

dbt project layering — the standard answer

staging (stg_) 1:1 with sources · rename/type/clean → intermediate (int_) business logic · joins · reusable blocks → marts (fct_/dim_) dimensional, business-facing, domain-organized

Semantic / metrics layer — hot topic

Problem it solves: "revenue" gets defined five different ways across BI tools. The layer centralizes metric and dimension definitions; BI tools query through it for governed, consistent metrics ("headless BI").

dbt Semantic Layer / MetricFlowCubeLookMLAtScaleMalloymetrics: simple/ratio/cumulative/derived

KPI governance: single source of truth for definitions, certified datasets, ownership, metric versioning, data contracts between producers and consumers.

Probes"Structure a dbt project." · "view vs table vs incremental — when each?" · "Implement SCD Type 2 in dbt." · "Design a star schema — what's your grain?" · "Build a semantic layer so finance and ops agree on revenue."
Readings — Analytics Engineering & dbt
  • dbt Labs (free) — "dbt Fundamentals" course + the "How we structure our dbt projects" guide. The course teaches the mechanics; the structure guide is the staging→marts answer interviewers expect, written by the people who invented the term "analytics engineer."
  • Kimball Group — the free "Dimensional Modeling Techniques" index pages. Authoritative source for grain, fact types, SCDs, conformed dimensions. Read the SCD and "declare the grain" entries closely.
  • dbt docs — Materializations, Incremental models, Snapshots. SCD2-via-snapshot and incremental strategies are the most-tested advanced topics; the docs are concise and exact.
  • dbt Semantic Layer / MetricFlow docs — the concepts page (semantic models, entities, measures, metrics). Learn the object model so you can speak the metrics-layer language even without a full build.
  • Book — The Data Warehouse Toolkit (Kimball), Ch. 1–5. If you read one modeling book, this is it. Skim for vocabulary; the worked examples cement grain.

04Data Quality, Reliability & Observability

Trust is the product. This is where reliability-minded consultants get repeat business.

The dimensions of data quality — recite these

AccuracyCompletenessConsistencyTimelinessValidityUniquenessIntegrity

Observability — the 5 pillars

FreshnessVolumeSchemaDistributionLineage

Be able to give a monitoring example for each (e.g., Freshness → alert if a table hasn't updated in 25h against a 24h SLA; Distribution → alert if null-rate or a column's mean shifts beyond a baseline).

Testing & validation frameworks

  • dbt tests (generic + singular), dbt_expectations, Great Expectations (suites, checkpoints, data docs), Soda, Monte Carlo (anomaly detection).
  • Assertion-based (explicit rules) vs anomaly-detection (ML/statistical baselines) — precision vs coverage/maintenance trade-off.

Reliability engineering (DataOps)

  • Data SLAs / SLOs / SLIs for pipelines (freshness, completeness).
  • Data contracts — schema + semantic agreements enforced at the producer boundary.
  • Circuit breakers — halt a pipeline on a failed quality gate to stop bad-data propagation.
  • Idempotency & reprocessing / backfills · quarantine / dead-letter for bad records.
  • Root cause analysis via lineage; data incident management.
Probes"Design a data quality framework for critical pipelines." · "5 pillars of observability — give a monitoring example for each." · "Where do you put a circuit breaker in an ELT flow?"
Readings — Data Quality & Observability
  • Monte Carlo — "The 5 Pillars of Data Observability" + their data-quality dimensions posts. They coined the 5-pillars framing; read it from the source so your vocabulary matches the field.
  • Great Expectations docs — the concepts tour + a profiling tutorial. Hands-on object model for assertion-based validation; you'll build a suite in the itinerary.
  • dbt docs — "Tests" + the dbt_expectations package README. The fastest path to test coverage on models; dbt_expectations ports GE-style checks into dbt.
  • Chad Sanderson — the "Data Contracts" essays (Substack / blog). The clearest practitioner case for contracts and producer-side enforcement; increasingly asked about in senior loops.
  • Google SRE Book (free) — the SLO / SLI chapter. Borrow the reliability vocabulary; reframing SRE for data pipelines reads as senior.

05Orchestration & Pipeline Design

How the pieces run, retry, and recover.

Orchestration concepts

  • DAGs, tasks, dependencies, scheduling.
  • Airflow (task-centric): operators, hooks, sensors, XComs, executors (Celery/Kubernetes), idempotent tasks, backfills, catchup, dynamic task mapping.
  • Dagster (asset-centric): software-defined assets, I/O managers, partitions, type checks, observability-first. Contrast with Airflow's task model.

Pipeline design patterns

batch / streaming / micro-batchidempotencyretriesexactly-once vs at-least-oncebackfillingwatermarksfan-in / fan-outorchestration vs choreography

Streaming vocabulary (even if batch-focused)

  • Kafka: topics, partitions, consumer groups, offsets, log compaction.
  • Stream processing: windowing (tumbling/sliding/session), event-time vs processing-time, watermarks, exactly-once.
  • Lambda vs Kappa architecture (batch+stream vs stream-only).
Probes"Airflow vs Dagster — contrast the models." · "Make a pipeline idempotent and handle backfills." · "Where do watermarks matter?"
Readings — Orchestration
  • Dagster docs — "Software-Defined Assets" tutorial. The asset-centric model is the modern framing and runs cleanly on a Mac locally; this is your hands-on path.
  • Astronomer / Airflow docs — the "Core Concepts" + idempotency/backfill guides. Airflow is still the incumbent in most shops; know its task-centric vocabulary for interviews.
  • Confluent — "Apache Kafka 101" intro modules. Enough streaming vocabulary (partitions, offsets, consumer groups) to hold the conversation.
  • Book — Designing Data-Intensive Applications (Kleppmann), Ch. 11. The definitive treatment of stream processing, watermarks, and exactly-once if you want depth.

06Reporting / BI Modernization

Where the modernization work becomes visible to the business.

  • Live query vs import/extract (Power BI VertiPaq, Tableau extracts) — performance vs freshness.
  • Star schema for BI — why BI tools love conformed stars.
  • Pre-aggregation / aggregate tables / materialized views for dashboard speed.
  • Row-level security (RLS) and governed access.
  • Self-serve BI + semantic layer to prevent metric sprawl.
  • Report rationalization — consolidate, certify, deprecate dead reports (usage analytics).
Probe"Modernize 400 legacy SSRS/Excel reports — your approach?" → rationalize → consolidate onto governed marts → semantic layer → certified dashboards → deprecate by usage.
Readings — BI Modernization
  • Microsoft Learn — Power BI "Star schema guidance" + import vs DirectQuery. Vendor-authored but genuinely the clearest explanation of the extract-vs-live trade-off and why stars matter for BI.
  • Metabase docs — the modeling + permissions guides. Your local BI tool in the itinerary; the permissions guide covers RLS hands-on.
  • Looker / LookML intro — the "what is a semantic model" overview. The original governed-metrics-in-BI approach; useful contrast to the dbt Semantic Layer.

07Architecture & System Design

The cross-cutting reasoning that separates an implementer from an architect.

Foundational trade-offs

  • OLTP vs OLAP; normalization vs denormalization.
  • CAP theorem & PACELC (consistency / availability / latency).
  • Consistency models: strong vs eventual.
  • The eternal triangle: cost vs performance vs freshness — you're always trading among these.
  • Push vs pull; schema-on-read vs schema-on-write.

Scalability & distributed-systems basics

partitioning / shardingreplicationhorizontal vs vertical scalingdata skew / hot partitionsshuffle costbroadcast vs shuffle joinscaching / result cachingbackpressuredead-letter queues

Governance, security, lineage

  • Catalog & metadata: Unity Catalog, Collibra, Alation, OpenMetadata, DataHub.
  • Lineage (column- vs table-level), impact analysis.
  • RBAC / ABAC, column/row-level security, masking / tokenization.
  • Compliance: GDPR, CCPA, HIPAA, SOX, PCI — know which your target industries (healthcare, insurance, finance) care about.
  • Data contracts, schema registry, MDM; FinOps (tagging, workload isolation, usage monitoring).

Readings — Architecture & System Design
  • Book (the one to own) — Designing Data-Intensive Applications (Kleppmann), Ch. 1–6, 9. The reference for replication, partitioning, consistency, CAP. Nothing else explains the trade-offs as clearly.
  • Book — Fundamentals of Data Engineering (Reis & Housley). The lifecycle + architecture survey tailored to exactly this consulting domain; pairs with DDIA's depth.
  • DataHub / OpenMetadata docs — the "what is data lineage / a data catalog" intros. Governance vocabulary (column lineage, impact analysis) you'll be expected to use.
  • Cloud FinOps Foundation — the "FinOps principles" overview. Cost-as-a-first-class-constraint framing; lets you talk spend like an architect, not just an engineer.

08How Design Interviews Are Scored

The meta-skill. Memorizing terms without this loop won't pass a senior round.

  1. Gather requirements first — volume, latency, consumers, SLAs, budget, sources — before designing.
  2. Declare assumptions & constraints out loud.
  3. Reason about trade-offs — name the alternatives and justify; there's rarely one right answer.
  4. Grain / source-of-truth discipline (the analytics-engineering signal).
  5. Failure handling — what breaks, how you detect, how you recover (the reliability signal).
  6. Cost awareness — cloud spend is a first-class constraint.
  7. Incremental delivery / migration safety — rollback, parallel run, phased cutover.
  8. Governance & quality baked in, not bolted on.
The reusable answer shaperequirements → assumptions → declare grain / source-of-truth → propose design → name trade-offs & alternatives → failure & recovery → cost. Run every prompt through this loop.

High-frequency prompts to rehearse

  • Migrate a legacy on-prem EDW to a cloud lakehouse — walk me through it.
  • Design the analytics layer for a company moving off spreadsheets.
  • Validate that a migration moved data correctly. (your sweet spot)
  • Design a data quality / observability framework for critical pipelines.
  • Implement SCD Type 2 in dbt — explain the mechanics.
  • Build a semantic layer so finance and ops agree on "revenue."
  • Design an incremental dbt model handling late-arriving data.
  • Reduce Snowflake/BigQuery cost without hurting SLAs.
